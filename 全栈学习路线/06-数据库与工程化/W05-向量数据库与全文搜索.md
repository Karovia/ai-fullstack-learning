# 第 06 章 · 第 05 周 · 向量数据库与全文搜索

## 一、为什么这一周很关键

LLM 时代，**"知识库 + 向量检索 + 重排"几乎是所有 AI 应用的底座**（RAG）。同时全文搜索也没死——它和向量是互补的两条腿。这周把检索打通。

---

## 二、为什么传统搜索不够

**类比**：
- "字符匹配"是按字面找，"皇上"和"皇帝"匹配不上
- "向量搜索"是按**意思**找，把"皇上""皇帝""陛下"映射到高维空间相近的点

```
查询："修车要多少钱"
传统全文搜索会漏掉的相关文档：
  - "汽车维修报价"
  - "保养费用怎么算"
  - "4S 店收费贵吗"
```

向量搜索能找到这些"语义相近"的内容。

## 三、嵌入向量 (Embedding) 是什么

把任意文本（图片、音频也行）丢进**嵌入模型**，得到一个固定维度的浮点数组：

```
"我喜欢猫"  →  [0.23, -0.11, 0.87, ...] (1536 维)
"我爱猫咪"  →  [0.21, -0.10, 0.85, ...] (1536 维) ← 几乎相同
"今天天气好" → [-0.51, 0.92, 0.04, ...]            ← 完全不同
```

**距离衡量"相似度"**：

| 度量 | 公式 | 用途 |
|---|---|---|
| **cosine** | 1 - 夹角余弦 | 文本最常用 |
| **dot** | 内积 | 嵌入归一化后等价 cosine，更快 |
| **L2 / Euclidean** | 欧氏距离 | 图像、坐标 |

### 主流嵌入模型

| 模型 | 维度 | 特点 |
|---|---|---|
| OpenAI `text-embedding-3-small` | 1536 / 可降维 | 便宜稳定 |
| OpenAI `text-embedding-3-large` | 3072 | 效果最强，贵 |
| `bge-m3`（智源） | 1024 | 开源最强多语言 |
| `bge-large-zh-v1.5` | 1024 | 中文专精 |
| Voyage AI `voyage-3` | 1024 | 闭源效果突出 |
| Cohere `embed-v3` | 1024 | 多语言均衡 |

```python
from openai import OpenAI
client = OpenAI()
resp = client.embeddings.create(model="text-embedding-3-small", input=["你好"])
vec = resp.data[0].embedding   # 1536 个 float
```

## 四、ANN 算法（怎么在亿级向量里快速找最近）

精确搜索 (k-NN) 复杂度 O(N)，亿级数据撑不住。**ANN（近似最近邻）** 牺牲一点点召回率换速度。

| 算法 | 思路 | 特点 |
|---|---|---|
| **HNSW** | 多层小世界图 | 工业首选，召回高速度快，内存大 |
| **IVF** | 聚类分区，先选区再搜 | 适合超大规模，可与 PQ 组合 |
| **PQ**（乘积量化） | 向量压缩成短码 | 内存小，召回略低 |
| **IVF-PQ** | 二者组合 | FAISS 经典 |

记住：**HNSW 是 90% 场景的最佳起点。**

## 五、pgvector：在 PostgreSQL 里存向量

最适合"已经在用 PG，不想再加一个新组件"的项目。

### 安装

```bash
docker run -d --name pgvec -p 5432:5432 -e POSTGRES_PASSWORD=devpass pgvector/pgvector:pg16
```

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE docs (
  id     bigserial PRIMARY KEY,
  title  text,
  body   text,
  embedding vector(1536)
);

-- HNSW 索引（推荐，PG 16+）
CREATE INDEX ON docs USING hnsw (embedding vector_cosine_ops);

-- 或 IVFFlat（旧）
-- CREATE INDEX ON docs USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
```

### 距离运算符

| 运算符 | 距离 |
|---|---|
| `<->` | L2 |
| `<#>` | 负内积（越小越相似） |
| `<=>` | cosine 距离（**文本用这个**） |

### 检索

```sql
-- Top 10 最相似
SELECT id, title, 1 - (embedding <=> $1) AS sim
FROM docs
ORDER BY embedding <=> $1
LIMIT 10;
```

```python
import numpy as np
from pgvector.asyncpg import register_vector

await register_vector(conn)
q_vec = await embed("修车要多少钱")
rows = await conn.fetch("SELECT id, title FROM docs ORDER BY embedding <=> $1 LIMIT 10", q_vec)
```

**配合元数据过滤**：

```sql
SELECT id, title FROM docs
WHERE author_id = $2 AND created_at > now() - interval '30 days'
ORDER BY embedding <=> $1 LIMIT 10;
```

## 六、专业向量库 Qdrant

向量量大（>千万）或要分布式时，专业库更合适。

```bash
docker run -d --name qdrant -p 6333:6333 -p 6334:6334 qdrant/qdrant
```

```python
from qdrant_client import AsyncQdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct

q = AsyncQdrantClient(host="localhost", port=6333)

await q.create_collection(
    collection_name="docs",
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE),
)

await q.upsert("docs", points=[
    PointStruct(id=1, vector=vec1, payload={"title": "..."}),
])

hits = await q.search(
    "docs", query_vector=q_vec, limit=10,
    query_filter={"must": [{"key": "author_id", "match": {"value": "alice"}}]},
)
```

### 各家对比

| 工具 | 强项 | 弱项 |
|---|---|---|
| **pgvector** | 事务、SQL、最简单 | 单机、超大规模略吃力 |
| **Qdrant** | Rust 性能高，过滤强大 | 需独立服务 |
| **LanceDB** | 文件级嵌入式（类似 SQLite） | 生态新 |
| **Chroma** | 上手最快，本地原型 | 生产化弱 |
| **Weaviate** | 模块化，自带嵌入 | 复杂 |
| **Milvus** | 超大规模 | 运维重 |
| **Pinecone** | 全托管 | 闭源、贵 |

**选型建议**：原型用 Chroma；中型项目用 pgvector；专业向量服务用 Qdrant；超大规模用 Milvus。

## 七、全文搜索方案

### PostgreSQL FTS（够用且免费）

```sql
ALTER TABLE docs ADD COLUMN tsv tsvector
  GENERATED ALWAYS AS (
    to_tsvector('simple', coalesce(title,'') || ' ' || coalesce(body,''))
  ) STORED;

CREATE INDEX idx_docs_tsv ON docs USING gin(tsv);

SELECT id, title, ts_rank(tsv, q) AS rank
FROM docs, plainto_tsquery('simple', '修车 价格') q
WHERE tsv @@ q
ORDER BY rank DESC LIMIT 10;
```

中文需要分词扩展，如 `pg_jieba`、`zhparser`。

### Meilisearch（轻量、上手快）

```bash
docker run -d --name meili -p 7700:7700 \
  -e MEILI_MASTER_KEY=devkey \
  getmeili/meilisearch:v1.10
```

```python
from meilisearch_python_sdk import AsyncClient
m = AsyncClient("http://localhost:7700", api_key="devkey")
idx = m.index("docs")
await idx.add_documents([{"id": 1, "title": "...", "body": "..."}])
res = await idx.search("修车", limit=10)
```

特点：开箱即用、毫秒响应、自动 typo 容错、支持中文。

### Elasticsearch（功能最强、最重）

适合超大规模、复杂聚合、日志场景。学习曲线陡，但企业里很常见。

### Typesense

类似 Meilisearch，开发体验略不同；二选一即可。

### BM25 算法

全文搜索打分核心。简化理解：**词频 × 逆文档频率 × 文档长度归一**。

- 词在文档里出现越多 → 分数越高
- 词在所有文档里越罕见 → 分数越高
- 文档越短 → 单次命中权重越高

## 八、混合检索（向量 + BM25 + Reranker）

**单一检索的问题**：
- 向量召回："语义近"但可能漏关键词
- 全文召回：精确关键词但抓不到同义

**Hybrid Search 流程**：

```
查询  ──┬──→ 全文搜索 (BM25)  → Top 50
        └──→ 向量搜索 (cos)   → Top 50
              ↓
          合并去重 → Top 100
              ↓
          Reranker 重排 (cross-encoder, 慢但准)
              ↓
          Top 10 返回 LLM
```

合并打分常用 **RRF (Reciprocal Rank Fusion)**：

```
score(d) = Σ 1 / (k + rank_i(d))    # k 通常取 60
```

### Reranker 模型

| 模型 | 特点 |
|---|---|
| **Cohere Rerank v3** | 闭源效果好，API 调用 |
| **bge-reranker-v2-m3** | 开源 SOTA，多语言 |
| **bge-reranker-large** | 更小可本地跑 |

```python
from sentence_transformers import CrossEncoder
ce = CrossEncoder("BAAI/bge-reranker-v2-m3")
pairs = [[query, doc.body] for doc in candidates]
scores = ce.predict(pairs)
ranked = sorted(zip(candidates, scores), key=lambda x: -x[1])[:10]
```

**经验**：召回 top 50，重排后 top 10，效果远好于单一向量召回 top 10。

## 九、文档解析与切片

### 解析器

| 工具 | 强项 |
|---|---|
| **PyMuPDF (fitz)** | 速度快，PDF 文本/表格 |
| **Unstructured.io** | 一站式，支持几十种格式 |
| **Docling** (IBM 开源) | 复杂版面/表格识别强 |
| **LlamaParse** | LlamaIndex 出品 |

### 切片策略

| 策略 | 说明 | 适合 |
|---|---|---|
| **固定字符** | 每 500 字一段 | 简单测试 |
| **递归字符** | 按 `\n\n → \n → 。 → ' '` 优先级切 | 通用 |
| **Markdown 标题** | 按 `#`/`##` 切，保留层级 | 文档站 |
| **语义切片** | 句子嵌入相似度变化点切 | 文学/演讲 |
| **小切片 + 大上下文** | 切小检索，命中后取大段返回 LLM | RAG 进阶 |

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500, chunk_overlap=80,
    separators=["\n\n", "\n", "。", "！", "？", " ", ""],
)
chunks = splitter.split_text(long_doc)
```

`overlap`（重叠）防止把关键句切散，通常 chunk_size 的 10-20%。

## 十、本周实战：100 篇 markdown 文章混合检索系统

**步骤**：

1. 准备 100 篇 markdown（你的笔记 / 技术博客均可）
2. 用 Docling 或 PyMuPDF 解析 → 用递归切片切到 500 字
3. 用 `text-embedding-3-small` 生成嵌入
4. 存到 pgvector，建 HNSW 索引
5. 同时把每个 chunk 写入 Meilisearch
6. 写检索函数：
   ```python
   async def hybrid_search(query, top_k=10):
       q_vec = await embed(query)
       vec_hits  = await pg_search(q_vec, 50)
       text_hits = await meili_search(query, 50)
       merged    = rrf_merge(vec_hits, text_hits)
       reranked  = await rerank(query, merged, top_k)
       return reranked
   ```
7. 评测：自己写 20 个 query + 期望相关文档，跑 Recall@10、MRR
8. 暴露 FastAPI `/search?q=...`，前端简单 UI 演示
