## 二、索引到底是什么

**类比**：一本 1000 页的书，你想找"PostgreSQL"出现在第几页。
- 没目录 → 翻 1000 页（**Seq Scan / 全表扫描**）
- 有目录 → 看目录 5 秒定位（**Index Scan**）

**代价**：目录占纸（占空间）、改页码要更新目录（写慢）。

### B-tree 索引图解

```
                     [50]
                    /    \
              [20,40]    [70,90]
              / | \      / | \
            ... 叶子节点（双向链表，方便范围扫描） ...
```

- 自平衡，所有叶子深度一样
- 等值、范围、排序、前缀模糊（`a%`）都能用
- **后缀模糊（`%a`）走不了 B-tree**

### 索引类型一览

| 类型 | 适合场景 |
|---|---|
| **B-tree**（默认） | 等值、范围、排序、`a%` 前缀 |
| **Hash** | 仅等值，速度略快，但功能弱很少用 |
| **GIN** | 数组、jsonb、全文搜索 tsvector |
| **GiST** | 几何、范围类型、全文搜索 |
| **SP-GiST** | 非平衡数据（IP、电话） |
| **BRIN** | 超大表 + 物理顺序写入（日志、时序） |

```sql
-- 普通
CREATE INDEX idx_user_email ON users(email);

-- 复合
CREATE INDEX idx_post_author_created ON posts(author_id, created_at DESC);

-- 唯一
CREATE UNIQUE INDEX uq_users_email ON users(lower(email));

-- 部分索引（只索引一部分行，瘦身）
CREATE INDEX idx_active_users ON users(created_at) WHERE is_active;

-- 表达式索引
CREATE INDEX idx_lower_email ON users(lower(email));

-- jsonb GIN
CREATE INDEX idx_meta ON users USING gin (metadata);
```

### 复合索引"最左匹配"

`(a, b, c)` 索引能加速：
- `WHERE a = ?`
- `WHERE a = ? AND b = ?`
- `WHERE a = ? AND b = ? AND c = ?`
- `WHERE a = ? ORDER BY b`

**不行**：`WHERE b = ?` 单独查（跳过 a），`WHERE c = ?`。

### 覆盖索引（Index-Only Scan）

```sql
CREATE INDEX idx_cover ON posts(author_id) INCLUDE (title, created_at);
```

`INCLUDE` 把数据"塞进"索引，查询不用回表。

### 该不该加索引？

加：高频查询条件、外键、JOIN 列、ORDER BY 列。
**别加**：
- 低基数列（性别、布尔）单独加意义不大
- 频繁更新的列
- 小表（<1 万行）
- 索引超过 5 个的表（写性能严重下降）

## 三、EXPLAIN ANALYZE：透视引擎的眼睛

```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM posts WHERE author_id = 'xxx' ORDER BY created_at DESC LIMIT 10;
```

输出关键字段：

| 关键字 | 含义 |
|---|---|
| **Seq Scan** | 全表扫描，红灯 |
| **Index Scan** | 走索引，绿灯 |
| **Index Only Scan** | 覆盖索引，蓝灯（最快） |
| **Bitmap Index Scan + Bitmap Heap Scan** | 命中很多行时的两步法 |
| **Nested Loop** | 一边小另一边有索引 |
| **Hash Join** | 一边构建哈希表 |
| **Merge Join** | 双方有序时合并 |
| **Sort** | 排序，注意是否走 disk |
| `cost=` | 估算代价 |
| `actual time=` | 实际耗时 |
| `rows=` | 实际行数 |
| `Buffers: shared hit=` | 缓存命中 |

**读法**：先看最里层的 Scan、再看 Join、最后看 Sort/Limit。注意 estimate 和 actual 偏差大说明统计信息过时，跑 `ANALYZE 表名`。

## 四、慢查询抓凶手

```sql
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
```

`postgresql.conf` 加 `shared_preload_libraries = 'pg_stat_statements'` 重启。

```sql
SELECT query, calls, mean_exec_time, total_exec_time
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

### 真实案例：5s → 50ms

**场景**：博客后台首页，"我所有文章 + 每篇评论数"。

```sql
SELECT p.id, p.title,
  (SELECT count(*) FROM comments c WHERE c.post_id = p.id) AS cnt
FROM posts p
WHERE p.author_id = $1
ORDER BY p.created_at DESC
LIMIT 20;
```

**问题**：相关子查询每行执行一次，5000 篇文章 → 5000 次 count，5 秒。

**EXPLAIN 看到**：comments 表 Seq Scan 5000 次。

**优化 1**：comments 表 `post_id` 加索引 → 1 秒（仍慢，count 5000 次）。

**优化 2**：改 LEFT JOIN + GROUP BY，**只对 LIMIT 后的 20 篇 count**：

```sql
WITH my_posts AS (
  SELECT id, title, created_at FROM posts
  WHERE author_id = $1 ORDER BY created_at DESC LIMIT 20
)
SELECT p.*, coalesce(c.cnt, 0) AS cnt
FROM my_posts p
LEFT JOIN (
  SELECT post_id, count(*) AS cnt FROM comments
  WHERE post_id IN (SELECT id FROM my_posts)
  GROUP BY post_id
) c ON c.post_id = p.id;
```

→ **50ms**。

## 五、事务 ACID

| 字母 | 含义 | 例子 |
|---|---|---|
| **A** Atomicity | 要么全成要么全败 | 转账两条 UPDATE 必须一起 |
| **C** Consistency | 不破坏约束 | 外键、CHECK 始终成立 |
| **I** Isolation | 并发互不影响 | 你看不到我未提交的修改 |
| **D** Durability | 提交后不丢 | 断电也在 |

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;        -- 全部生效
-- 或 ROLLBACK; 全部撤销
```

### 隔离级别 4 种 + 三类异常

| 级别 | 脏读 | 不可重复读 | 幻读 |
|---|---|---|---|
| Read Uncommitted | ✗ 可能 | ✗ | ✗ |
| **Read Committed**（PG 默认） | ✓ 不会 | ✗ | ✗ |
| **Repeatable Read**（PG 实际是 SI） | ✓ | ✓ | ✓（PG 实现挡掉了） |
| Serializable | ✓ | ✓ | ✓ |

- **脏读**：读到别人未提交的修改
- **不可重复读**：同事务两次读同一行结果不同（被别人 UPDATE）
- **幻读**：同事务两次按条件查数量不同（被别人 INSERT/DELETE）

切换：`SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;`

## 六、锁

### 行锁

```sql
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;   -- 行锁
-- 其他事务对这行的 UPDATE/SELECT FOR UPDATE 阻塞
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

`FOR SHARE` 是共享锁（多个事务可同时持有，互不阻塞读，阻塞写）。

`FOR UPDATE SKIP LOCKED`：跳过别人锁住的行，常用于队列消费者。

`FOR UPDATE NOWAIT`：拿不到立刻报错。

### 表锁

`LOCK TABLE x IN ACCESS EXCLUSIVE MODE;` —— 一般别手动用，DDL 会自动加。

### 死锁

A 事务先锁 row1 再锁 row2；B 事务先锁 row2 再锁 row1 —— 互等。

PostgreSQL **自动检测死锁**（默认 1 秒），杀掉其中一个事务并报错 `deadlock detected`。

**避免**：所有事务按**统一顺序**加锁（如按 id 升序）。

## 七、MVCC：PG 不阻塞读的秘密

每行有 `xmin`（创建它的事务 id）和 `xmax`（删除它的事务 id）。

UPDATE 不是原地改，而是**写一个新版本**，旧版本留着供其他事务读，等没人引用了 VACUUM 回收。

**好处**：读不加锁、读不阻塞写、写不阻塞读。
**代价**：表会膨胀（dead tuples），需要 `VACUUM`、`VACUUM FULL`、`autovacuum`。

```sql
-- 观察膨胀
SELECT relname, n_live_tup, n_dead_tup
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;

VACUUM ANALYZE posts;   -- 回收 + 更新统计
```

### 实战 2：并发更新库存防止超卖

```python
# 错误写法：先 SELECT 再 UPDATE，并发下会超卖
async def buy_wrong(item_id):
    stock = await db.fetchval("SELECT stock FROM items WHERE id=$1", item_id)
    if stock > 0:
        await db.execute("UPDATE items SET stock = stock - 1 WHERE id=$1", item_id)

# 正确写法 1：原子 UPDATE
async def buy_atomic(item_id):
    rows = await db.execute(
        "UPDATE items SET stock = stock - 1 WHERE id=$1 AND stock > 0",
        item_id,
    )
    return rows == 1   # 是否抢到

# 正确写法 2：行锁
async def buy_lock(item_id):
    async with db.transaction():
        stock = await db.fetchval(
            "SELECT stock FROM items WHERE id=$1 FOR UPDATE", item_id
        )
        if stock <= 0:
            return False
        await db.execute("UPDATE items SET stock = stock - 1 WHERE id=$1", item_id)
        return True
```

用 `asyncio.gather` 起 1000 个并发请求验证库存不会出错。
