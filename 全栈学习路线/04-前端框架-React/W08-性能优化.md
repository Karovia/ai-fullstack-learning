## 一、React 渲染机制（先建立心智模型）

一次更新分三步：

1. **Render 阶段**：调用组件函数，生成新虚拟 DOM（VDOM）。
2. **Reconcile 阶段**：和旧 VDOM diff，标记要变更的真实 DOM。
3. **Commit 阶段**：一次性把所有变更刷到 DOM。

类比：
- VDOM = 设计稿；
- Reconciler = 装修队列长（决定哪面墙要砸）；
- Commit = 工人（真去砸）。

### 为什么会重渲染（这是面试题）
1. 自己 `setState`（即使值一样也会触发，除非 React 判断 `Object.is` 相等）。
2. 父组件重渲染 → 子组件默认跟着渲（**不是 props 变才渲，是父渲就跟着渲**）。
3. `useContext` 的值变了 → 所有消费组件都渲。

> **重渲染 ≠ 重画 DOM**。Render 是 JS 跑一遍，DOM 只在差异处改。但 Render 多了照样卡。

## 二、React DevTools Profiler（必学）

1. 装 React DevTools 浏览器扩展。
2. 打开 Profiler 面板 → 点 ⏺ 录制 → 操作页面 → 停止。
3. 看 **Flamegraph**：横条越长 = 渲染越久。
4. 看 **Ranked**：按耗时排序的组件。
5. 点组件可看"为什么渲染了"（"Why did this render?" 要在 Settings 里勾 "Record why each component rendered"）。

**判断方法**：
- 一次交互超过 16ms（60fps 一帧）= 掉帧。
- 同一个组件在你以为它不应该渲的时候渲了 = 有优化空间。

## 三、React.memo（用对了才有用）

```tsx
const Child = memo(function Child({ value }: { value: number }) {
  console.log('Child render')
  return <div>{value}</div>
})
```

`memo` 的规则：**props 浅比较一样 → 跳过 render。**

### 陷阱 1：传对象/数组字面量
```tsx
<Child data={{ a: 1 }} />          // ❌ 每次都是新对象，memo 失效
<Child onClick={() => doX()} />     // ❌ 每次都是新函数
```

### 陷阱 2：children 是 JSX
```tsx
<Memoized><span>x</span></Memoized> // children 每次新引用
```

### 解法：useMemo / useCallback 让引用稳定

```tsx
const data = useMemo(() => ({ a: 1 }), [])
const onClick = useCallback(() => doX(), [])
<Child data={data} onClick={onClick} />
```

### 真的需要吗？

| 场景 | memo 值 |
| --- | --- |
| 子组件本身渲染很轻（一个 `<div>`） | 不必，memo 比较都比渲染贵 |
| 子组件渲染昂贵（图表、表格、Markdown） | 值得 |
| 子组件在长列表里 | 值得 |

**默认不用 memo**。Profiler 看到瓶颈再加。

## 四、useMemo / useCallback 真正用法

```tsx
// ✅ 计算昂贵：sort/filter 大数组
const sorted = useMemo(() => bigList.sort(...), [bigList])

// ✅ 引用要稳：传给 memo 子组件 / 当 useEffect 依赖
const config = useMemo(() => ({ url, headers }), [url])
useEffect(() => { fetch(config.url) }, [config])

// ❌ 包基础类型：useMemo(() => 1+1, [])  没意义
// ❌ 每个函数都包 useCallback：本身有开销
```

### React 19 / React Compiler

> **未来你不用手写这些**。
> React Compiler（实验性）会**自动**给组件 / 值加 memo。届时 `useMemo` 和 `useCallback` 几乎只在边缘场景才需要手写。

## 五、长列表：虚拟滚动

> 1 万条 `<li>` 全渲染 = 卡死。**只渲染可视区那 20 条**。

### TanStack Virtual

```tsx
import { useVirtualizer } from '@tanstack/react-virtual'

function List({ items }: { items: string[] }) {
  const ref = useRef<HTMLDivElement>(null)
  const v = useVirtualizer({
    count: items.length,
    getScrollElement: () => ref.current,
    estimateSize: () => 40,
    overscan: 5,
  })
  return (
    <div ref={ref} style={{ height: 400, overflow: 'auto' }}>
      <div style={{ height: v.getTotalSize(), position: 'relative' }}>
        {v.getVirtualItems().map(row => (
          <div key={row.key} style={{
            position: 'absolute', top: 0, left: 0, width: '100%',
            transform: `translateY(${row.start}px)`, height: row.size
          }}>
            {items[row.index]}
          </div>
        ))}
      </div>
    </div>
  )
}
```

`react-window` 是更老的替代品，API 简单一些。

## 六、Code Splitting：React.lazy + Suspense

按需加载组件，**首屏 bundle 小一半**：

```tsx
import { lazy, Suspense } from 'react'
const Editor = lazy(() => import('./Editor'))   // 单独 chunk

<Suspense fallback={<Spinner />}>
  <Editor />
</Suspense>
```

路由级懒加载（Next 默认就帮你做了，自己用 react-router 也能做）：
```tsx
const About = lazy(() => import('./pages/About'))
```

## 八、Tree-shaking 与 ESM

打包工具能砍掉**没被用到的导出**，前提：
1. 用 ESM (`export` / `import`)，不要 CommonJS。
2. `package.json` 标 `"sideEffects": false`（或精确列出有副作用的文件）。
3. 不要写 `import * as _ from 'lodash'`，会全打进来。

## 九、图片优化（next/image）

```tsx
import Image from 'next/image'
<Image src="/cover.jpg" alt="封面" width={1200} height={630}
       priority sizes="(max-width: 768px) 100vw, 800px" />
```

自动做：
- WebP / AVIF 输出；
- 多尺寸 srcset（按设备分辨率给不同图）；
- `loading="lazy"`（`priority` 关掉，用于首屏 LCP）；
- 防止 CLS（强制要求 width/height 或 fill）。

## 十、字体（next/font）

```tsx
import { Inter } from 'next/font/google'
const inter = Inter({ subsets: ['latin'], display: 'swap' })
<html className={inter.className}>...
```

构建时把字体下载到自己服务器 → **0 网络请求到 Google + 0 闪烁**。

`font-display: swap` 让文字先用降级字体展示，字体好了再切，避免空白。

## 十二、防抖 / 节流 / 懒加载组件

```tsx
// 搜索框防抖
const [q, setQ] = useState('')
const debounced = useDeferredValue(q)         // React 内置
useEffect(() => { search(debounced) }, [debounced])

// 滚动节流（也可手写 + rAF）
import { throttle } from 'es-toolkit'
const onScroll = useMemo(() => throttle(() => { ... }, 100), [])

// IntersectionObserver 懒加载组件
const ref = useRef<HTMLDivElement>(null)
const [show, setShow] = useState(false)
useEffect(() => {
  const io = new IntersectionObserver(([e]) => e.isIntersecting && setShow(true))
  if (ref.current) io.observe(ref.current)
  return () => io.disconnect()
}, [])
return <div ref={ref}>{show && <Heavy />}</div>
```

## 十三、SSR 优化

- 把慢的部分用 `Suspense` 包起来，让首屏先到，慢的后到（streaming）。
- 数据获取放在最近的 Server Component 里，避免 prop drilling。
- 静态能 SSG 就 SSG，最快。

## 十四、Web Vitals（衡量体感）

| 指标 | 含义 | 目标 |
| --- | --- | --- |
| **LCP** Largest Contentful Paint | 最大元素出来的时间 | < 2.5s |
| **INP** Interaction to Next Paint | 交互到下一帧 | < 200ms |
| **CLS** Cumulative Layout Shift | 累计布局偏移 | < 0.1 |
| **FCP** First Contentful Paint | 首个内容出现 | < 1.8s |
| **TTFB** Time To First Byte | 首字节时间 | < 800ms |

线上看：Chrome DevTools → Lighthouse；或装 [`web-vitals`](https://github.com/GoogleChrome/web-vitals)，上报到自己后台。

```ts
import { onLCP, onINP, onCLS } from 'web-vitals'
onLCP(v => console.log('LCP', v.value))
onINP(v => console.log('INP', v.value))
onCLS(v => console.log('CLS', v.value))
```

## 实战：找一个慢页面，优化到 60fps

**步骤**：
1. 选一个之前写的列表/表格页面，故意造 5000 条数据。
2. 用 Profiler 录一段交互，截图保存"优化前"。
3. 找瓶颈：
   - 列表 → 加虚拟滚动。
   - 子组件全渲染 → memo + 稳定引用。
   - 包体大 → bundle analyzer 定位 + 懒加载。
4. 再录一次，对比帧时间。
5. 写一篇博客《我把这个页面从 X ms 优化到 Y ms》：
   - 截图前后 Flamegraph
   - 列出 5 个最有效的改动
   - 反思 1 个走过的弯路

## 常见错误

| 现象 | 原因 | 解法 |
| --- | --- | --- |
| 加了 memo 还是渲 | props 里有内联对象/函数 | 用 useMemo / useCallback 稳引用 |
| 列表虚拟滚动滚得卡 | item 高度估错 | 用动态测量 `measureElement` |
| 懒加载白屏闪 | 没有 Suspense fallback | 加骨架 |
| CLS 飙高 | 图片没设宽高 | 显式 width/height 或 aspect-ratio |
| `useMemo([dep])` 漏依赖 | 闭包陈旧值 | 老老实实把每个引用都列进去（ESLint 会警告） |
