## 1. Redis 五大类型

```
String   "user:1:name" -> "alice"
List     "queue:emails" -> [a, b, c]    （LPUSH / BRPOP 做队列）
Hash     "user:1" -> {name: alice, age: 20}
Set      "user:1:tags" -> {python, web}
ZSet     "leaderboard" -> {alice: 99, bob: 87}  （SCORE 排序）
```

进阶：Stream（持久化消息流）、HyperLogLog（基数统计）、Bitmap、Geo、Pub/Sub（W04 用过）。

Docker 起一份：

```bash

docker exec -it redis redis-cli
```

常用命令：`SET / GET / EXPIRE / DEL / KEYS（生产禁用，用 SCAN）/ INCR / DECR / TTL / EXISTS`。

## 2. 缓存模式

- **Cache-Aside（旁路）**：业务读时先查缓存，没就查 DB 写回缓存。**最常用**。
- **Read-Through**：缓存层负责回源，业务只跟缓存说话（如 Redis 模块 / CDN）
- **Write-Through**：写时同时写 DB 和缓存
- **Write-Back**：先写缓存后异步刷 DB；性能高但崩了丢数据

Cache-Aside 范式：

```python
async def get_article(id: int):
    key = f"article:{id}"
    cached = await r.get(key)
    if cached: return json.loads(cached)
    article = await db_get(id)
    if article:
        await r.set(key, json.dumps(article), ex=300)
    return article

async def update_article(id, payload):
    await db_update(id, payload)
    await r.delete(f"article:{id}")     # 失效 > 更新（避免不一致窗口）
```

## 3. 缓存三大灾难

- **缓存穿透**：查不存在的 key，每次打到 DB。**解法**：给"找不到"也缓存空值（短 TTL，如 60s）；或前置 BloomFilter
- **缓存雪崩**：大量 key 同时过期，DB 被冲垮。**解法**：TTL 加随机抖动、热点 key 永不过期 + 后台异步刷新
- **缓存击穿**：单个热点 key 失效瞬间，N 个请求同时回源。**解法**：互斥锁（只允许一个请求回源）、逻辑过期（不真正过期，到期后异步重算）

互斥锁示意：

```python
async def get_with_mutex(id: int):
    key, lock = f"article:{id}", f"lock:article:{id}"
    cached = await r.get(key)
    if cached: return json.loads(cached)
    if await r.set(lock, "1", nx=True, ex=10):
        try:
            article = await db_get(id)
            await r.set(key, json.dumps(article), ex=300)
            return article
        finally:
            await r.delete(lock)
    else:
        await asyncio.sleep(0.1)
        return await get_with_mutex(id)
```

## 4. 限流四算法

每种都画一遍才记得住：

- **固定窗口**：每分钟最多 N 次。简单。**缺点**：窗口边界容易翻倍流量。
- **滑动窗口**：把窗口切成 M 小格，按时间偏移。Redis ZSET 实现：score = timestamp，过期的 ZREMRANGEBYSCORE。
- **漏桶**：固定速率出水，进水溢出就丢。强制平滑。
- **令牌桶**：固定速率加 token，请求消耗 token；token 满了停止增加。**允许短时突发**——业界默认。

Redis 滑动窗口：

```python
async def rate_limit(key: str, limit: int, window_s: int) -> bool:
    now = time.time()
    pipe = r.pipeline()
    pipe.zremrangebyscore(key, 0, now - window_s)
    pipe.zcard(key)
    pipe.zadd(key, {f"{now}-{uuid4()}": now})
    pipe.expire(key, window_s)
    _, count, _, _ = await pipe.execute()
    return count < limit
```

slowapi 内部就是这套思路。生产建议：直接用 [redis-rate-limit-py](https://github.com/upstash/ratelimit-py) 或 `slowapi` 替代手写。

## 5. 分布式锁

```python
async def acquire(name: str, ttl: int = 10) -> str | None:
    token = uuid4().hex
    if await r.set(f"lock:{name}", token, nx=True, ex=ttl):
        return token
    return None

async def release(name: str, token: str):
    # Lua 保证 check-and-del 原子
    lua = """if redis.call("get", KEYS[1]) == ARGV[1] then
              return redis.call("del", KEYS[1]) else return 0 end"""
    await r.eval(lua, 1, f"lock:{name}", token)
```

更强一致性场景用 **Redlock**（多 Redis 节点过半数获得才算成功），社区有现成实现 `redlock-py` / `pottery.Redlock`。但 Martin Kleppmann 写过著名"Redlock 真的安全吗"——分布式锁本质是难题，能不用就不用。

## 6. 任务队列

为啥要：长任务（发邮件 / 视频转码 / AI 调用）不能堵 HTTP 线程；要重试、要可观测、要削峰。

**Python 选择**：
- **ARQ**：基于 Redis + asyncio，轻、现代
- **Celery**：老牌全功能，配置复杂
- **Dramatiq**：折中

**Node 选择**：BullMQ（Redis 队列）。

ARQ 示例：

```bash
uv add arq
```

```python
# tasks.py
from arq.connections import RedisSettings

async def send_email(ctx, to: str, subject: str, body: str):
    print(f"[email] {to} / {subject}")
    # 真实发送...

class WorkerSettings:
    functions = [send_email]
    redis_settings = RedisSettings(host="localhost", port=6379)
    max_jobs = 50
    job_timeout = 60

# 启动 worker：arq tasks.WorkerSettings
```

入队：

```python
from arq import create_pool
from arq.connections import RedisSettings

pool = await create_pool(RedisSettings())
job = await pool.enqueue_job("send_email", "a@b.com", "Hi", "Welcome")
```

ARQ 自带：重试、定时（cron-like）、并发限制、死信队列。

## 8. 结构化日志

`print` 是新手；线上要 JSON 一行一条，含 trace id / 用户 id / 耗时。

```bash
uv add loguru structlog
```

最简（loguru）：

```python
from loguru import logger
import sys, json

logger.remove()
logger.add(sys.stdout, serialize=True)  # serialize 输出 JSON

logger.bind(request_id="abc-123", user_id=42).info("article.created", article_id=99)
```

输出：

```json
{"text":"...","record":{"extra":{"request_id":"abc-123","user_id":42,"article_id":99},"level":{"name":"INFO"},"message":"article.created",...}}
```

把 W02 的 `RequestIDMiddleware` 中 `rid` 通过 ContextVar 传进 logger，每条日志自带 request_id：

```python
from contextvars import ContextVar
request_id_var: ContextVar[str] = ContextVar("request_id", default="-")

def patcher(record):
    record["extra"]["request_id"] = request_id_var.get()

logger.configure(patcher=patcher)

# 中间件里 set
request_id_var.set(rid)
```

## 11. OpenTelemetry 入门

OTel 是统一三大支柱（traces / metrics / logs）的开源标准，避免被某家厂商锁死。

```bash
uv add opentelemetry-api opentelemetry-sdk \
       opentelemetry-instrumentation-fastapi \
       opentelemetry-exporter-otlp
```

```python
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

trace.set_tracer_provider(TracerProvider())
trace.get_tracer_provider().add_span_processor(
    BatchSpanProcessor(OTLPSpanExporter(endpoint="http://localhost:4317"))
)
FastAPIInstrumentor.instrument_app(app)
```

后端可接：Jaeger / Tempo / Honeycomb / Grafana Cloud。
